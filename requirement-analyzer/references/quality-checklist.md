# 审核 Agent 质量检查清单

> 审核 Agent 在 Step 7 使用的结构化检查清单。
> 按顺序逐项验证，每项的验证结果记录在审核结论中。

---

## 1. 事实核查（每条 req-*.md 的 §4）

- [ ] 每个"已完成"声明有至少一个 commit hash
- [ ] commit hash 可通过 `git show <hash>` 验证存在
- [ ] commit 的 diff 范围与声明一致
- [ ] 每个文件路径引用可被 Read 或 get_code_snippet 验证
- [ ] 每个行号引用对应位置的代码片段与文档描述一致
- [ ] 每个"未完成"声明有 `git log -G "<pattern>"` 0 命中的证据
- [ ] 验证项 Checklist 中的每项证据可复现

## 2. 完整性核查（每条 req-*.md）

- [ ] wiki 文档中的所有子需求点在分析文档中有对应分析
- [ ] §2 的锚点覆盖了需求涉及的所有关键代码位置
- [ ] §3 的调用链覆盖从 API 入口到持久化/外部调用的完整路径
- [ ] §5 的改动文件清单覆盖了所有锚点引用的文件
- [ ] §6 的风险清单 ≥ 3 项
- [ ] 涉及数据库变更时有 DDL 说明（或明确标注无需 DDL）
- [ ] 涉及外部 API 变更时有关联接口章节

## 3. 一致性核查（跨文档）

- [ ] SUMMARY.md 的完成度判定与各 req-*.md 的 §4 结论一致
- [ ] SUMMARY.md 中引用的文件路径可解析（文件存在）
- [ ] 各文档间的术语使用一致（同一概念不使用不同名称）
- [ ] 序号一致（SUMMARY.md 中的 #N 与 req-NN-*.md 的序号对应）
- [ ] 迭代归类正确（需求在正确的迭代目录下）

## 4. 风险覆盖核查

- [ ] 向后兼容性已考虑（API 参数变更、返回值结构变更）
- [ ] 数据库迁移/回填已考虑（DDL 变更的风险和回退方案）
- [ ] 缓存失效策略已考虑（涉及缓存数据的变更）
- [ ] 并发安全已考虑（涉及共享状态的修改）
- [ ] 空值/边界条件已考虑（null 兜底、空列表处理）
- [ ] 灰度/回退方案已提及
- [ ] 外部依赖变更已标注（Feign 调用方、MQ 消费方）
- [ ] 前端影响已注明（即使本次只关注后端）

## 5. 行动顺序核查

- [ ] SUMMARY.md 中的优先级分类（P0/P1/P2/P3）合理
- [ ] P0 项确实是阻塞发布或存在安全/数据风险的
- [ ] P1 项之间没有未标注的依赖关系
- [ ] 依赖外部团队（DBA、产品、前端）的项已标注
- [ ] 预估工作量与实际改动规模基本匹配（明显低估或高估的需标注）

## 6. 格式规范核查

- [ ] 代码片段有正确的语言标记（```java）
- [ ] 表格对齐，无不完整行
- [ ] 文件路径格式统一（代码文件用相对路径，引用用绝对路径）
- [ ] 行号格式统一（`lines N-M` 或 `line N`）
- [ ] 结论性文字（YES/NO/PARTIAL）用粗体突出
- [ ] 修改日志表格格式正确
